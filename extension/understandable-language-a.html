Understandable language (A)

Understandable language: Provide clear language in instructions,   labels, navigational elements, and error messages that require a response to continue, so that the following are true:

 For instructions,  labels, and navigational elements:

  • Simple tense: Use the present tense and active voicing. (See exceptions for different context and language.) 
  • Simple and clear words: Use words or phrases that are most frequently used for the current context unless this will result in a loss of meaning or clarity.  This includes not using abbreviations words and phrases unless they are the common form to refer to the concept for beginners. Where word frequencies are known for the context, they can be used.
  • Double negatives are not used
  • Literal text: Metaphors and non-literal text are not used or can be automatically replaced via an easy to set user setting. All meaning must be retained when non-literal text are replaced.

 Also on controls:

  • The words on controls and labels  identify the element's function.

 Also on instructions:

  • Each step in instructions are identified and concrete wording is used.
Exceptions:
  • There are times when a passive voice or other tense can be clearer. Other voicing may be used when it has been shown via user test to be easier to understand, more friendly or appropriate.
  • In languages where present tense and active voicing does not exist or is  not clearer in the language of the content. In this case use the tense and voicing that is clearest for the content.
  • The present tense is not required when describing or discussing past or future events.
  • If the writing style is an essential part of the main function of the site, such as a  a game or literary work or teaching new terms.
  • Where less common words are found in user testing to be easier to understand for the audience (user testing should included people with cognitive disabilities that could be in the target audience - need further definition ) 
  • The writing style items may be replaced for a location or type of content were user testing has shown a more effective writing style to aid comprehension for people with cognitive disabilities. Such as for content written in a specific language.
  • The content will be penalized for not conforming to a given writing style (such as a CV, dissertation or PhD proposal).

Description

The intent of this success criteria is that people can understand and use navigational element, user interface and instructions. Clear language on all content is an important accessibility principle. However if the users does not understand words and terms in these critical areas the whole application or website often becomes unusable.

A real life example is of a user with mild dementia trying to use an app to turn on the heating and air conditioning unit. The menu item for selecting heat or air conditioning is labeled "mode". The user does not know that mode refers to heat or air and can not use the whole unit because of this one term.

In this real life example (reported by a task force member) a visitor turned on the air conditioner and did not turn it off when they left. The weather became a bit cooler and the user, who could not turn on the heat because of the language used, became hypothermia and needed emergency treatment.

People with dementia have impaired short term memory and difficulty remembering new information, so learning and remembering new terms can be impossible. However if the interface uses familiar terms and design it is fully usable. Not being able to use these applications mean that more people require live in help and lose their independence.

In another example, many task force members can not use github because the terms used are not the normal terms used for the functions (such as push instead of upload).

It should be noted that the restrictions on scope makes it practical from the content providers perspective and the exception ensure that it is widely applicable.

Benefits

This supports those who have reading difficulties, language disabilities and some visual perceptual difficulties. It can include people with intellectual disabilities, Receptive Aphasia and acquired dyslexia as well as those with general cognitive learning disabilities, where long sentences and complex words can cause difficulties. This supports those who have Dementia and/or acquire cognitive disabilities as they Age.

Additionally it supports persons with general cognitive learning disabilities, where long sentences and complex words can cause difficulties.

COGA Techniques

Testability

The success criteria is testable if each of the bullet points are testable. If the content fails any bullet points it is not conforment to this success criteria. If it passes ALL of the bullet points it is conformant ‎

Bullet points:

  • Simple tense: Use the present tense and active voicing. (See exceptions for different context and language.) 

The tense and voicing is a fact and hence verifiable. We also expect natural language processing algorithm will be able to conform this automatically with reasonable accuracy.

Testing for exceptions:

If present tense and active voicing have not been used the tester will need to confirm if one of the exceptions are relevant. If an exception is not relevant and present tense and active voicing have not been used then the content fails this success criteria

  • Simple and clear words: Use words or phrases that are most frequently used for the current usage unless this will result in a loss of meaning or clarity. Where word frequencies are known for the context, they can be used.  This includes not using abbreviations words and phrases unless they are the common form to refer to the concept for beginners.

Even languages with a small number of users have published lists of the most frequent words (such as Hebrew). Algorithm exist that calculate these lists for new languages or for specific contexts. Testing content against these word lists can be done manually but it is expected it will be a natural language processing testing tool by the time this goes to CR. (It is already integrated into a tool by IBM.)

Testing for exceptions is as discussed above.

  • Double negatives are not used

Use of negatives is a fact and hence this is verifiable. It is assumed natural language processing tool will also test for this. Testing for exceptions is as discussed above.

 

  • Literal text: Metaphors and non-literal text are not used or can be automatically replaced via an easy to set user setting. All meaning must be retained when non-literal text are replaced.

Non-literal text and metaphors can be identified when the meaning of the sentence is something other then the meaning of the individual words. This is human testable and cognitive computing algorithms can test for this as well.

If the text is not literal, then the tester must confirm that personalization and an easy user setting enables it to be replaced such that all meaning is retained.

  • The words on controls and labels  identify the elements  function.

 This can be tested by identifying the function of the control and checking it is in identified in the label.

  • Each step in instructions is identified and concrete wording is used

This is human testable by completing the instructions literally and confirming that the effect is correct.
